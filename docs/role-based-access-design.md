# Role-Based Access Architecture (RBAA) Design

## Status

Design only — **not yet implemented**. `src/api/main.py` currently exposes
`/analyze` with no authentication or authorization; anyone who can reach the
API can upload and analyze financial data. This document specifies the
target model referenced as "Future Enhancement" scope in the README and
flagged as a gap in `docs/design_review.md` (§1, §5).

## Purpose

The Finance Intelligence Platform is designed using Role-Based Access
Control (RBAA) principles. Each user is assigned permissions according to
their responsibilities, ensuring financial information is accessed only by
authorized personnel while reducing operational risk. This document defines
the roles, their permissions, how report approval and alert routing map to
those roles, and where enforcement would live once authentication exists.

Financial reporting systems process sensitive business information. Separating
responsibilities between analysts, managers, executives, and administrators 
reduces operational risk, supports governance, and aligns with enterprise 
security best practices.

## Roles and Permissions

| Role                   | Responsibilities                                                          | System Access                                                              |
| ---------------------- | -------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Financial Analyst      | Upload financial CSVs, review own generated reports                        | Upload files, view own reports and their status                             |
| Finance Manager        | Review KPIs, approve reports before distribution, receive High/Critical alerts | View all reports, approve/reject pending reports, export reports, receive notifications |
| Finance Director (CFO) | Review approved executive summaries and business trends                    | View approved reports and dashboards, access historical reporting, receive Critical alerts only |
| System Administrator   | Maintain workflows, troubleshoot failures, manage integrations             | Full access, including Audit Trail, Processing Failures, and system configuration |

### Financial Analyst

- Upload CSV files.
- View reports generated from their own uploads.
- Cannot approve or distribute reports.
- Cannot modify workflows or system configuration.
- Cannot view other analysts' uploads unless also granted Finance Manager access.

### Finance Manager

- View all reports across analysts.
- Approve or reject a report before it is distributed (see
  [Approval Workflow](#approval-workflow-and-report-status) below).
- Receive High and Critical alerts.
- Review AI-generated summaries.
- Export reports.

### Finance Director

- View only reports that have passed Finance Manager approval.
- Receive Critical alerts only (not High).
- View executive dashboards and historical reporting.
- Review business trends across reporting periods.
- Cannot approve or edit reports — read-only at the executive level.

### System Administrator

- Manage the FastAPI service and its configuration.
- Maintain n8n workflows and credentials.
- Monitor the Audit Trail.
- Resolve processing failures and dead-lettered executions.
- Configure integrations (OpenAI, Airtable, Slack/Email) and rotate credentials.
- Implicitly has read access to everything, for troubleshooting — this should
  itself be logged, since it's the broadest grant in the system.

## Approval Workflow and Report Status

`docs/assumptions_and_open_questions.md` lists "Who approves reports before
they're distributed?" as an open question, and `docs/design_review.md` (§1)
flags that the current pipeline sends AI-generated summaries straight to
stakeholders with no human sign-off. This RBAA design assumes that gap is
closed as follows:

1. A report generated by the pipeline enters status `pending_approval` and
   is visible only to the Financial Analyst who uploaded it and to Finance
   Managers.
2. A Finance Manager reviews the KPIs, AI summary, and any flagged anomalies,
   then transitions the report to `approved` or `rejected`.
3. Only `approved` reports become visible to the Finance Director and trigger
   stakeholder notifications.
4. `rejected` reports notify the originating Analyst with the Manager's
   reason and do not go out to notification channels.

This status field does not exist yet in `src/models/financial_schema.py` or
in the storage layer — it is a prerequisite for RBAA to mean anything beyond
"who can view what," since without it there is nothing for a Manager's
approval permission to act on.

## Alert Routing by Severity

| Severity | Recipients                          | Channel(s)      |
| -------- | ------------------------------------ | ---------------- |
| Info     | Financial Analyst (own uploads only) | In-app / email    |
| High     | Finance Manager                      | Email + Slack     |
| Critical | Finance Manager, Finance Director    | Email + Slack     |
| System failure | System Administrator          | Slack (ops channel) |

This reconciles the two places severity was previously mentioned
inconsistently: the permissions table said Finance Managers receive
"High/Critical" while Directors receive only "Critical," which is preserved
here as the authoritative mapping. Recipient lists themselves (which Slack
channel, which distribution group) are still undefined — `design_review.md`
§5 calls this out as a leak risk and it remains an open item, not something
this document can resolve alone.

## Enforcement Mechanism

RBAA is only as real as its enforcement point. This system has three places
permissions would need to be checked, none of which exist today:

- **FastAPI (`src/api/main.py`)** — the `/analyze` endpoint (and any future
  report-retrieval endpoints) needs an auth dependency (e.g.
  `Depends(get_current_user)`) that resolves an authenticated identity to a
  role, mirroring how `get_llm_client` is already wired as a FastAPI
  dependency. Until this exists, the roles in this document are not
  enforced by the API layer at all.
- **n8n workflows** — notification and approval branching (who a Slack
  message or email goes to) needs to read the report's status and the
  recipient's role rather than a static distribution list, or role changes
  require editing the workflow itself.
- **Airtable** — if Airtable remains the MVP store (see `docs/decisions.md`,
  which currently only has an ADR for n8n — Airtable's own ADR is still
  outstanding per `design_review.md` §6), role enforcement there is limited
  to Airtable's own per-base collaborator permissions, which are coarser
  than the roles above and cannot distinguish "Financial Analyst" from
  "Finance Manager" within a single base without workarounds (separate
  views at best, not real row-level security).

**No identity provider is chosen yet.** This document assumes users are
authenticated somehow (SSO, Airtable collaborator identity, or a simple
API-key-per-role scheme for the MVP) but does not pick one. That decision
should get its own ADR in `docs/decisions.md` before implementation starts,
since it determines how roles get assigned to users in the first place.

## Out of Scope for V1

- Fine-grained, per-report or per-company row-level permissions
  (multi-company support is already a Future Enhancement elsewhere in the
  docs, and would push RBAA toward per-tenant roles).
- Self-service role assignment — role changes are expected to be a System
  Administrator action for V1, not a workflow.
- Delegated/temporary access (e.g., an Analyst covering for a Manager).
- MFA / SSO — deferred until an identity provider is chosen.

## Open Questions

- Which identity provider (if any) issues the authenticated identity that
  roles attach to?
- Does report `status` (pending/approved/rejected) live in Airtable, or does
  it require a store with real transactional guarantees, given
  `design_review.md`'s concerns about Airtable as an audit-trail store?
- Should System Administrator read access to financial report contents be
  scoped down (e.g., metadata/logs only, not report contents) to reduce the
  blast radius of that role, or is full access required for troubleshooting?
